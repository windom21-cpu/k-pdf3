## 各項目の詳細

### #1 workspace 保持ポリシー ADR + 手動お掃除機能 🔴 最重要

**目的**: workspaces フォルダの無制限膨張 (実測 7.4 GB、35 個/日 ≒ 年 40 GB 超ペース) を止め、ディスク逼迫の事故を予防する。

**背景・前提知識**:
- PDF を開くたびに元 PDF を丸ごと BLOB 複製した workspace が作られる (ADR-0007 `userData/workspaces/` 集中保管 + `index.db` fingerprint 索引)
- 掃除機構は現状 `src/main/sidecar-sweep.js` のみで、対象は `*.kpdf3.source.pdf` サイドカーだけ。**`.kpdf3` 本体を消す仕組みは存在しない**
- **ADR-0026 の制約 (最重要)**: 確定保存で旧 workspace が「編集可能マスター」として predecessor 参照されるようになった。**predecessor として参照されている workspace を消すと「編集に戻す」が壊れる**。`sidecar-sweep.js` に将来の掃除で predecessor を消すなという注記が既にある。ADR-0026 も「孤児 workspace 掃除は将来 opt-in」と明記
- 実測では 200 KB 未満 (ほぼ「開いただけ」) が 62 個しかない = **大半は何らかの編集を持つ**。「空だけ消す」では容量問題は解決しないので、保持期間ベースの判断が必要

**対応方針 (推奨)**:
1. **ADR-0027 を起草してユーザー承認を得てから実装** (保存データの削除ポリシーはレイヤ/schema に匹敵する重要判断)
2. MVP は**手動実行のみ** (ツールメニュー「ワークスペースの整理…」)。自動削除はしない (法律実務データなので誤削除リスクを取らない)
3. 削除候補の判定案 (ADR で提示する叩き台):
   - (a) predecessor として参照されている → **無条件で保持**
   - (b) overlay ゼロ・削除/挿入/回転/並び替えなし (= 開いただけ) → 即候補
   - (c) 編集ありは「最終アクセスから N ヶ月」で候補 (N=3/6/12 をユーザーに選ばせる。デフォルト案 6)
4. UI: 候補の件数・合計容量・ファイル名リストをプレビュー → 確認モーダル → 実行。`index.db` (workspace-registry) のエントリも同時に削除して索引の整合を保つ
5. 完全削除かごみ箱 (`shell.trashItem`) か → ごみ箱推奨 (誤削除の保険)。ADR で確認

**ユーザー確認が必要な点**: 保持期間 N / ごみ箱 or 完全削除 / 「編集あり」も消してよいか (Dropbox 上のフラット PDF が残るので理論上は再取込可能だが、overlay の編集可能状態は失われる) 。

**完了条件**: 掃除実行後に (a) 既存案件が正常に開く (b) 確定版の「編集に戻す」が生存 (c) `index.db` に删除済エントリが残らない。`test/sidecar-sweep.test.mjs` の並びに保持判定の単体テストを追加。

✅ **完了 (2026-07-05)**: v2.0.12-beta.2 を実機適用し、確認項目 ①スキャン ②移動後に既存案件が開く ③確定版「編集に戻す」生存 ④ごみ箱に .kpdf3 ⑤再スキャンで候補ほぼ 0 — **全て想定どおり、問題なし** (ユーザー実機確認)。

実装記録 (2026-07-05):
- ADR-0027 起草 → ユーザー承認済 (編集ありも期間候補 / N デフォルト 3 ヶ月 / ごみ箱移動)
- 実装: `src/main/workspace-cleanup.js` (判定コア、read-only スキャン) + IPC 2 本 (`workspace-cleanup-scan/execute`) + ツールメニュー「ワークスペースの整理…」+ 98 風ダイアログ。テスト 31 件 (編集種別×判定のテーブル駆動 + predecessor 保護 fixpoint + 実ファイル end-to-end) を electron-runner に追加、全 pass
- **実データ dry-run (read-only) の発見**: 1,811 個中 **1,338 件 (2.90 GB) が「開いただけ」**で N に関係なく候補。レビュー時の「開いただけ 62 個」は 200KB という代理指標の誤り (workspace は元 PDF を丸ごと含むため開いただけでも大きい)。編集あり 473 件は全て 3 ヶ月以内アクセスで保持。スキャン所要 ~45 秒 (WSL /mnt/c 経由、実機はより速い想定)
- 残: 実機での (a)(b)(c) 確認 + ごみ箱移動の動作確認 (shell.trashItem は Windows 実機でのみ本番挙動)
- **v2.0.12-beta.2 として配信済 (2026-07-05、tag 796f585、CI success、k-pdf3-releases に installer + latest.yml 公開)**。実機確認項目: ① ツール → ワークスペースの整理… でスキャン (候補 ~1,300 件 / ~2.9GB 想定) ② 移動後に既存案件が開く ③ 確定版の「編集に戻す」生存 ④ ごみ箱に .kpdf3 が入っている ⑤ もう一度スキャンして候補 0 付近

### #3 パスワード保護 PDF の平文化警告 (+ 将来の再暗号化)

**目的**: 現状、パスワード付き PDF を開くと qpdf `--decrypt` で復号版が workspace に保存され、**保存/書き出しするとパスワードが外れた PDF がディスク (=Dropbox) に置かれる** (HANDOVER §15.4 (a))。法律実務では「相手方がパスワード付きで送ってきた文書が平文で同期される」情報管理事故になり得るため、まず利用者が気づける状態にする。

**対応方針**:
- **MVP**: 復号して開いた直後 (実ユーザーパスワードを入力した場合) に 98 風モーダルで一度警告:「このファイルを保存・書き出しするとパスワード保護は外れます」。二度と出さないチェックは付けない (案件ごとに意識すべき情報)
- **将来 (opt-in)**: 保存/書き出し時に開いた時のパスワードで qpdf `--encrypt` 再暗号化するオプション。そのために **MVP 時点で workspace metadata に `source_was_encrypted` フラグを残しておく** (パスワード自体は保存しない — 保存するなら別途設計判断が要る)
- ADR-0025 (パスワード復号、起草待ち) を #7 で書く際にこの警告仕様と再暗号化の将来案を含めると一石二鳥

**実装ポイント**: 復号経路は `src/main/main.js` の `kpdf3:open-pdf-file` 内 `pdfIsEncrypted` → qpdf `--decrypt` (パスワード stdin)。入力モーダルは `src/renderer/dialogs.js` の `customPasswordPrompt`。open の戻り値に `wasDecrypted` を足して renderer 側で警告表示するのが素直。

**完了条件**: パスワード付き PDF を開く→警告表示→保存→出力が従来通り平文 (回帰なし)。権限制限のみ/空パスワード PDF (プロンプトなしで開くもの) に警告を出すかは要判断 (推奨: ユーザーパスワード入力時のみ警告)。

🚧 **MVP 実装済 (2026-07-05)、実機確認待ち**:
- 警告: `openPdfPath` (renderer.js) で**実ユーザーパスワードを入力して復号した場合のみ**、開き終わった直後に 98 風モーダル (customConfirm, OK のみ) で「保存・書き出しするとパスワード保護の無い PDF が作成される」旨を警告。推奨どおり権限制限のみ/空パスワード PDF は対象外。「次から表示しない」なし。全 open 経路 (開くボタン/最近使った/D&D/新タブ) は openPdfPath に集約されているので 1 箇所で網羅
- 挙動メモ: 復号ゲートは毎オープンで on-disk の暗号化バイトに走るため、実パスワード PDF は**再オープンでも再入力→警告が毎回出る** (案件ごとに意識すべき情報という方針と整合)
- 将来布石: 復号を伴う取込時に workspace metadata `source_was_encrypted="1"` を記録 (`Workspace.markSourceWasEncrypted()` / `sourceWasEncrypted()`、パスワード自体は保存しない)。既存 workspace も再オープン時に後追いで記録される
- テスト: `test/source-encrypted-flag.test.mjs` (5 件、electron-runner に追加)。`npm test` 全 suite pass
- 残: 実機確認 — パスワード付き PDF を開く→警告表示→保存→出力が従来通り平文 (回帰なし)

✅ **完了 (2026-07-06)**: **v2.0.12-beta.3 として配信済み** (tag push → CI success、installer + latest.yml 公開)。実機確認はユーザー判断により**実運用上で確認する**扱いとし、本項はクローズ。問題があれば都度対応。

- 検証 1 (ゲート単体): 各状態で `isCopy` 判定が「完全未編集のみ true」になること
- 検証 2 (end-to-end 抜き取り): 代表ケースを実際に export → mupdf で再読込し、編集が PDF バイトに焼かれていること

**完了条件**: `npm test` に組み込まれ全 pass。既知 3 バグを故意に再導入すると fail することを確認 (ミューテーションで有効性検証)。

### #5 Electron 41 EOL (2026-08-25) の方針決定

**目的**: Electron 41 のサポートが **2026-08-25 で終了** (あと約 7 週間)。42 は better-sqlite3 の prebuild 未対応 (PR #1470 取り下げ、issue #1474/#1475 オープン) でまだ上げられない。期限までに 42 が通らなかった場合の方針を**今のうちに決めておき**、慌てて悪手 (自前ビルド強行等) を打たない。

**判断材料**:
- 月次監視 routine (`trig_014hyr1dE1yVZZWf23J1PRZN`、毎月 1 日 09:05 JST) が better-sqlite3 の 42 対応を自動判定し、揃えば 42 化 PR を準備する — この枠組みは維持
- **推奨: (a) EOL 一時容認 + 監視継続**。根拠: ローカル完結アプリでリモートコンテンツを開かない、CSP + window.open/遷移拒否の hardening 済 (β.145)、攻撃面が狭い。Electron 38 EOL 時 (3 ヶ月容認) の前例あり
- 代替 (b) better-sqlite3 自前ビルド: CI に native build が入り Win/Mac/Linux 3 OS 分の複雑化・保守負担が発生。見合わない
- (a) の場合も「容認は better-sqlite3 対応まで。対応が出たら次の patch/β で即 42 化」と条件付きにする

**作業**: ユーザーに (a)/(b) を提示 → 決定を ADR-0004 の §更新節に追記 (HANDOVER 更新は #7 と同時でよい)。コード変更なし。

### #9 確定版ステータスの常時可視化 + 「編集に戻す」文言再考

**目的**: ADR-0026 で PDF に「確定版 (マスターあり) / マスター切れ / 下書き」という状態が生まれた。開いた瞬間のヒントは実装済だが、タブを多数開く業務では「今どっちを触っているか」を**常時**見えるようにすると迷いが減る。また「編集に戻す」は Undo (元に戻す) と紛らわしい懸念が β トライアル検証項目 (4) に挙がっている。

**対応方針 (β トライアルの実機所感を聞いてから着手)**:
- タイトルバー or ステータスバーに常時表示 (例: `契約書.pdf 〔確定版・戻せます〕`)。タブラベルへのマーク付与も検討 (dirty マークと同居できるか要デザイン)
- 文言案: 「編集に戻す」→「**確定を解除して編集**」等、Undo と衝突しない表現。ツールバーボタンの tooltip も併せて
- 状態は open 戻り値の `hasEditableMaster` / `masterMissing` (v2.0.12-beta.1 実装済) で判定可能。表示だけの追加で保存コアには触らない
- ADR-0026 の UI 追補として同 ADR に追記 (新 ADR は不要)

**着手条件**: ユーザーの β トライアル所感 (紛らわしかったか・常時表示が欲しいか) を聞いてから。stable 昇格前の β.2 に載せるか昇格後の次版かはその時に判断。
