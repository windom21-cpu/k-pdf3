# ADR-0026: 確定保存を可逆にする編集可能マスター（戻せる確定）

- 日付: 2026-07-01
- ステータス: **実装済（v2.0.12-beta.1 で実装 → β トライアルで実機確認 → v2.0.13 stable に収録）**
- 関連: ADR-0007（userData workspace 集中保管 + fingerprint 索引）、ADR-0008（exports BLOB 廃止 — 本 ADR はその「将来 opt-in で復活」条項の後継 + 版記録は別名保存運用が担う思想を継承）、ADR-0011（Save As workspace 切替）、HANDOVER §保存モデル（下書き／確定）

## Context

### ユーザーの実ワークフローと二律背反

法律実務で、受信 PDF にテキスト・吹き出しコメント・印影を足して使う。ユーザーは **後からテキストを直すことが多いので「下書き保存（編集可能として上書き）」を常用**している。一方で、**成果物を Dropbox から直接（Adobe 等で）閲覧することも多い**。

ここで現行の 2 モードがどちらも要求を満たさない：

| 保存モード | 元 PDF ファイル | 他アプリ / Dropbox から見える | 後から編集 |
|---|---|---|---|
| **下書き保存**（編集可能として上書き） | **一切触らない**（workspace のみ） | ❌ 何も反映されない | ✅ K-PDF3 で編集可能 |
| **確定保存**（画像として上書き） | 焼き込んで上書き | ✅ 反映される | ❌ 画像化され動かせない |

- 下書き常用 → **Dropbox から見ると編集が一切見えない**（実際にユーザーが遭遇：K-PDF3 では追加ページが見えるのに Dropbox 実体には無く、最終更新日時も変わっていなかった）。
- 確定すると見えるが **後から吹き出しを直せない**（＝ユーザーが下書きを選ぶ動機を潰す）。

### 現行アーキテクチャの確認（本 ADR の前提）

1. **workspace は「元 PDF バイトの SHA-256 fingerprint」で索引**される（ADR-0007、`workspace-registry.js`）。
2. **描画は workspace 内部に保持した元 PDF のコピー（`workspace.getSourceBytes()`）から行う**。ディスク上の実ファイルは「開くときに fingerprint を取る」「初回取り込み」にしか使わない（`main.js:1316,1424`）。
3. **確定保存の実挙動**：レンダリング → ディスクへ焼き込み書き出し → **焼き込みファイルを開き直す → 新しい fingerprint → overlay の無い新 workspace に切替**（`renderer.js:4180` 付近の再 open）。このとき **編集可能だった旧 workspace（pristine source + overlays）は削除されず userData に残る**（ただし通常の「開く」からは辿れない＝孤児化）。現行の掃除（β.147 `sidecar-sweep`）は兄弟 `.kpdf3` が生きている限り消さない。
4. **overlay は小さい**（テキスト・座標のみ）。容量を食うのは元 PDF バイト。挿入した外部 PDF/画像は `inserted_source_pdfs` が **SHA-256 で重複排除済**（`sqlite-store.js`）。
5. ADR-0008 で **exports の実体 BLOB は容量のため削除**（50MB×10回=550MB→50MB）、監査ログ（hash/size/timestamp/revisionId）のみ残す。**bit-identical な提出物の現物はユーザーの別名保存運用（ディスク上に版名付きで残す）で確保**する思想。「絶対保持要件が再浮上したら opt-in で ADR-00xx として復活」と明記済。

**要点**: 「編集可能」と「ファイルに反映」が両立しにくいのは本質的制約ではなく、確定保存が旧 workspace を孤児化して捨てているだけ。編集可能状態は workspace 側に丸ごと在り、**確定の直前に明示的に編集可能マスターとして温存すれば「戻す」ことができる**。

## Decision

**確定保存を破壊的操作でなくす。** 確定＝「配布用フラット PDF をディスクに書き出す ＋ その時点の編集可能状態を『編集可能マスター』として温存する」に変え、**「編集可能な状態に戻す」** で復元できるようにする。

保持形態は **単一マスター型（Model Y）** を採用する。

### 1. 確定保存の新しい定義

```
確定保存 = ① ディスクにフラット版を書き出す（Dropbox / Adobe で最新が見える）
           ② その時点の編集可能状態を「編集可能マスター」として温存（lineage で紐づけ）
戻す      = 編集可能マスターを開いて編集継続 → また確定でフラット版を更新
版の記録  = Dropbox 上のフラット PDF（日付 / 版名）が担う（ADR-0008 思想と一致）
```

### 2. 単一マスター型（Model Y）— バージョン履歴は持たない

- **ドキュメント 1 つにつき、編集可能マスターを 1 つだけ** K-PDF3 内（userData）に温存する。確定のたびに世代を積む**バージョン履歴は MVP では持たない**。
- 「戻す」で戻る先は **最新の編集可能状態**（＝直前の確定を押した瞬間の状態）。
- **過去世代への任意ロールバックは Dropbox 上のフラット PDF（版名付き）に委ねる**。これは ADR-0008 が確立した「bit-identical な現物は別名保存運用で確保」思想の踏襲であり、K-PDF3 内に版履歴を二重に持たない。
- バージョン履歴型（Model X: 世代スナップショット + 一覧 UI + 保持ポリシー N）は **将来の opt-in 拡張**とする（§解除条件）。

### 3. 「編集可能な状態に戻す」復元フロー

```
編集なしPDF
  → テキスト追加            （overlay = 編集可能）
  → 確定上書き保存          ─┬─ ディスクにフラット版を書き出し（Dropbox で見える）
                            └─ 編集可能マスターを温存（lineage 紐づけ）
  → 「編集可能な状態に戻す」  → マスターを開いて編集継続
```

**復元後の状態 = 確定を押した瞬間とまったく同じ「テキスト入り・編集可能」**。overlay は本物の編集レイヤとして戻るので、再度 動かす／直す／消す ができる（画像ではない）。

同時に成り立つ 2 つの真実：
- **ディスクのファイル（Dropbox/Adobe）**：テキストは見えるが焼き込み済み（そこでは動かせない）
- **K-PDF3 で「戻す」した状態**：テキストがまた編集可能

### 4. 同一性の紐づけ = workspace lineage（決定）

確定でファイルの fingerprint が変わるため、「このフラットファイルの編集可能マスターはどれか」は **workspace lineage** で結ぶ：

- 焼き込み後 workspace に `predecessor`（編集可能マスターへのポインタ）を持たせる。
- そのフラットファイルを K-PDF3 で開く（fingerprint → 焼き込み workspace）と、`predecessor` を辿って **「編集可能な状態に戻す」を提示**する。
- **パス非依存**なので Dropbox の移動 / リネームに強い。
- 外部で 1 バイトでも改変されると fingerprint が変わり lineage が切れる。この場合は黙って編集不能版にせず **「編集可能版が見つかりません」と明示**する。
- 却下：パス基準（Dropbox 移動で切れる）／PDF に GUID 埋め込み（綺麗な出力を汚す・他アプリが除去しうる）。

### 5. 容量戦略 — 元 PDF バイトは共有（dedup）、overlay 差分だけ

編集可能マスターの追加コストを **overlay 差分（通常 数十 KB）** に抑える。元 PDF バイト（大きい）は 1 度だけ保持して共有する。

| 実装方針 | 追加コスト | 採否 |
|---|---|---|
| workspace 丸ごとコピー | ＋PDF1個分（≒倍） | ❌ ADR-0008 が捨てた肥大の再来 |
| **元 PDF バイト共有 + overlay 差分のみ** | 数十 KB〜 | ✅ 単一マスターなので容量はほぼ据え置き |

例（10MB PDF にテキスト追加）: 丸ごと=+10MB、dedup=数十 KB。K-PDF3 は開いた時点で既に元 PDF の内部コピーを 1 つ持つ（既存の 1×）ので、本機能で **新規に増えるのは overlay 分だけ**。単一マスター型のため保持ポリシー N は不要。

### 6. 保存モード UI は 2 つのまま（決定）

- **下書き保存・確定保存の 2 モードは維持し、新しい保存ボタンは増やさない**。
- 「戻す」は保存モードではなく **別の復元アクション**（メニュー / ボタン）として置く → 保存の選択肢は今のまま 2 つ＝認知負荷を増やさない。
- 確定ダイアログの文言を改訂：「あとから動かせなくなります」→ **「あとで『編集可能な状態に戻す』で戻せます」**。

## Why この選択肢か

| 選択肢 | 採否 | 理由 |
|---|---|---|
| **A. 確定を可逆化（単一マスター + 戻す）（採用）** | ✅ | ディスクは普通のフラット PDF＝**Dropbox 同期で壊れない・互換性最強**、ファイル 1 個、保存コアの同定ロジックは触らず退避+復元を足すだけ、ADR-0008 が予告した opt-in 復活の正統な後継 + 版記録は別名保存運用に委ねる思想を継承 |
| B. 反映付き下書き（1ファイル・fingerprint 再登録で編集可能維持） | ❌ | fingerprint を同一 workspace に再登録＝**Dropbox 同期/複数台/競合コピーで紐づけが切れ、編集可能 workspace が孤児化**。しかも「見た目は反映済み」で誤信しやすい。保存コアへの回帰リスク最大 |
| C. 共有用コピーを自動生成（2ファイル） | △ | 低リスクだがファイルが 2 個・命名運用が要る。編集マスターが「見えない第2ファイル」になり運用が煩雑 |
| D. 現状維持（下書き/確定の 2 択のまま） | ❌ | 二律背反が解けない。確定の「後から動かせない」恐怖が残る |
| （粒度）バージョン履歴型 Model X | 将来 | 過去世代ロールバックは魅力だが保持ポリシー/一覧UI/管理が要る。版記録は ADR-0008 の別名保存運用で足りるため MVP は単一マスター、履歴は opt-in 拡張へ |

B の弱点（孤児化）が **Dropbox 環境でこそ顕在化する**のが決定的。A はディスクを普通の PDF に保つので同期に強い。

## Consequences

### 受け入れる trade-off

1. **編集可能マスターは「保存した PC の userData」にしか無い（ファイル/Dropbox には乗らない）。** 別 PC で開く・userData を消す・PC 交換・再インストールでは **残るのはフラット PDF だけで編集には戻せない**。フラットファイルを配る以上どの案でも同じ根本制約だが、本案では「戻せるのはこの PC だけ」と明示的に割り切る。macOS の Versions（OS/FS レベル・透過的）とは異なり、**K-PDF3 内・当該 PC 限定の編集マスター**である点を UI で伝える。
2. **過去世代へのロールバックは持たない（MVP）。** 「戻す」で戻れるのは最新の編集可能状態のみ。過去の確定版が要るときは Dropbox 上のフラット PDF（版名付き）を使う（ADR-0008 思想）。
3. **ADR-0008 の方針を部分的に上書き**：exports は監査メタのみ（据え置き）だが、**別系統で「編集可能マスター」を保持**する。bit-identical な提出物 BLOB を貯めるわけではない（overlay 差分 + 共有 source）ので、0008 が問題視した肥大とは別物。
4. **確定の意味が変わる**：「不可逆な焼き込み」→「配布用書き出し + 編集可能マスターの温存」。ダイアログ文言を改訂する（§Decision 6）。
5. **セキュア書き出し（qpdf メタ除去）との関係**：qpdf のメタ除去は **ディスクのフラット版だけに適用**。内部の編集可能マスターは常に完全な編集可能状態（メタ保持）を保つ。競合なし。
6. **掃除ロジックとの整合**：現状 確定は旧 workspace を孤児化して残すだけ（偶然生き残っている）。本案では **編集可能マスターを明示的に保護対象**にし、将来の「空/孤児 workspace 掃除」（未着手・低優先）で誤削除されないようにする。

### 影響範囲（想定）

- `src/domain/workspace.js`: 編集可能マスターの温存/復元 API（overlay 差分 + source 参照 dedup）、lineage 記録
- `src/backend/sqlite-store.js`: lineage / master 参照用スキーマ（source は共有参照、overlay/page 構造を保持）、migration
- `src/main/main.js` / `preload.cjs`: 確定前のマスター温存、`restore-editable-master` IPC、lineage（`predecessor`）配線
- `src/renderer/renderer.js`: 確定ダイアログ文言改訂、「編集可能な状態に戻す」UI（メニュー/ボタン）、lineage 切れ時の明示
- `src/main/sidecar-sweep.js` 付近: 編集可能マスター保護（誤掃除防止）
- **既存の印刷・回転・下敷き・下書き/確定の byte-copy ゲートには非干渉**（新経路の追加）

### 解除条件 / 将来

- **バージョン履歴型（Model X）**：過去世代ロールバック要件が出たら opt-in で上乗せ（世代スナップショット + 一覧 UI + 保持ポリシー N）。単一マスター型は dedup 済なので拡張は差分保持の延長。本 ADR の同一性/容量方針を土台に新 ADR で。
- **編集可能マスターを別 PC へ持ち運びたい**→ ADR-0007 §15.2 の export package(zip) 案に master を含める形で上乗せ（本 ADR は変更不要）。

## 実装時に詰める細部（設計フェーズ）

1. `predecessor` の保存先とスキーマ（焼き込み workspace 側 or registry 側）、および lineage 切れ検出の実装。
2. 巨大 PDF（β.134 external sidecar）での source 共有：BLOB でなく外部ファイル参照のケースの refcount / keep-alive（掃除から保護）。
3. 「編集可能な状態に戻す」の導線（ファイルメニュー / ツールバー / フラットファイルを開いた直後の誘導）と、マスターが無い/切れた場合の文言。
4. 確定を複数回行ったとき、マスターが最新確定状態で上書き更新される挙動の確認（単一マスターの一意性維持）。
5. 下書き保存との併用時の状態遷移（下書き→確定→戻す の一貫性）。

## 検証（実装時の実機確認項目・草案）

- 編集なしPDF → テキスト追加 → 確定上書き → Dropbox/Adobe で **テキストが見える**
- 続けて「編集可能な状態に戻す」→ **テキストが編集可能**（移動/変更/削除できる）で復元される
- 戻した後に再編集 → 再確定 → ディスクが更新され、また戻せる（マスターは最新で一意）
- 挿入ページ/回転/削除/並び替えを含む状態でも往復して構造が保たれる
- 容量：10MB 級 PDF でマスターを持っても userData 増分が overlay 差分規模に収まる（丸ごとコピーになっていない）
- 掃除処理で編集可能マスターが誤削除されない
- 外部でファイルを 1 バイト変えた後に開く → lineage 切れが UI で分かる（黙って編集不能版にならない）
- qpdf セキュア書き出しを ON にしてもマスターは編集可能なまま

## UI 追補（2026-07-10、REVIEW-2026-07 #9）

β トライアルの実機確認（往復・混在編集・文言とも問題なし）を経て stable 昇格。
あわせて、タブを多数開く業務で「今どちらを触っているか」を開いた直後の一言
（ws-status、次の操作メッセージで流れる）だけに頼らないよう、**ステータスバーに
文書状態を常時表示**する固定フィールドを追加（v2.0.13）：

- `確定版〔戻せます〕` — lineage があり［編集に戻す］が使える状態
- `確定版〔編集用データなし〕` — lineage ポインタはあるがマスターがこの PC に無い
- `下書き` — 通常の編集可能状態（上記以外）

状態判定は既存の `getEditableMasterInfo`（`hasEditableMaster` / `masterMissing`）を
そのまま使い、`refreshRestoreMasterUI`（open / タブ切替 / 確定 / 戻す / close の
全経路で発火済み）に表示更新を同乗させただけ。**表示のみの追加で保存コアは不変**。
「編集に戻す」のボタン文言は β トライアルで紛らわしさの報告が無かったため現状維持。
