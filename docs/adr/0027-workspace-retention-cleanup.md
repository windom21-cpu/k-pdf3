# ADR-0027: workspace 保持ポリシーと手動お掃除機能（ワークスペースの整理）

- 日付: 2026-07-05
- ステータス: **採用（2026-07-05 ユーザー承認済・実装着手）**
- 関連: ADR-0007（userData workspace 集中保管 + fingerprint 索引）、ADR-0008（ディスクフットプリント削減）、ADR-0026（戻せる確定保存 — predecessor 保護の根拠）、β.147 sidecar-sweep（掃除機構の先行例）

## Context

### 無制限膨張の実態（2026-07-05 実測）

PDF を開くたびに元 PDF を丸ごと BLOB 複製した workspace が `userData/workspaces/` に作られる（ADR-0007）が、**`.kpdf3` 本体を消す仕組みが存在しない**：

| 項目 | 実測値 |
|---|---|
| workspaces 合計 | **7.4 GB / 1,819 ファイル** |
| `.kpdf3` | 1,808 個 / 6.6 GB |
| 増加ペース | ≒35 個/日（年 40 GB 超ペース） |
| 「開いただけ」（200 KB 未満） | 62 個のみ = **大半は何らかの編集を持つ** |

既存の掃除は `sidecar-sweep.js`（β.147）のみで、対象は孤児 `*.kpdf3.source.pdf` サイドカーだけ。放置すればディスク逼迫は時間の問題。

### 制約（ADR-0026 由来・最重要）

確定保存された workspace は metadata `predecessor_workspace_id` で旧 workspace（編集可能マスター）を参照する。**predecessor として参照されている workspace を消すと「編集可能な状態に戻す」が壊れる**。`sidecar-sweep.js` にも「将来の掃除は predecessor を skip せよ」と明記済み。

### 「開いただけ」だけ消しても解決しない

実測どおり空 workspace は 62 個しかない。容量の主犯は「編集を持つが二度と開かない」workspace であり、**保持期間ベースの判定が不可欠**。ただし削除しても Dropbox 上のフラット PDF は残るため再取込は可能（失われるのは overlay の「編集可能な状態」のみ）。

## Decision

**手動実行のみの「ワークスペースの整理…」をツールメニューに追加する。自動削除は実装しない**（法律実務データなので誤削除リスクを取らない）。

### 1. 削除候補の判定規則（優先順に適用）

| 順 | 規則 | 判定 |
|---|---|---|
| 1 | いずれかの workspace から `predecessor_workspace_id` で参照されている（編集可能マスター） | **無条件で保持** |
| 2 | 現在アプリで開いているタブの workspace | **保持**（今回の整理対象外） |
| 3 | 「開いただけ」= overlay ゼロ・ページ削除/挿入/回転/並び替えなし | **即候補**（期間不問） |
| 4 | 編集あり | **最終アクセスから N ヶ月超で候補**。N は実行時に UI で選択（**3 / 6 / 12、デフォルト 3**）— 2026-07-05 ユーザー決定 |

- 「最終アクセス」= `index.db`（workspace-registry）の `updated_at` と `.kpdf3` ファイル mtime の**新しい方**。registry に行が無い孤児 workspace は mtime のみで判定
- predecessor 集合の収集は workspaces ディレクトリの全 `.kpdf3` を **read-only で開き** metadata を読む（手動実行なので 1,800 個規模の走査は許容。read-only なので WAL 汚染なし）

### 2. 削除の方式 — ごみ箱（2026-07-05 ユーザー決定）

- `shell.trashItem()` で **Windows のごみ箱へ移動**（完全削除はしない — 誤削除の保険）
- workspace に β.134 サイドカー（`{id}.kpdf3.source.pdf`）があれば**同時にごみ箱へ**
- `index.db` の該当行（`workspace_path` が削除対象を指す行）も**同時に DELETE** し、索引の整合を保つ

### 3. UI フロー（98 風・既存モーダル慣行に従う）

```
ツール → ワークスペースの整理…
  → 保持期間 N 選択（3/6/12 ヶ月、デフォルト 3）→ スキャン実行
  → プレビュー: 候補の件数・合計容量・一覧（ファイル名 / 最終アクセス / 編集有無 / サイズ）
     ＋「編集可能マスター ○ 件と開いているタブ ○ 件は保護されました」の表示
  → 確認モーダル（削除件数と容量を明示、既定ボタンはキャンセル）
  → 実行 → 結果表示（移動 ○ 件 / ○ MB をごみ箱へ）
```

### 4. やらないこと（スコープ外）

- **自動削除・起動時削除**: しない。手動のみ
- **バージョン履歴の管理**: ADR-0026 Model Y のまま（本 ADR は削除だけ）
- **バックアップ**: 別タスク（REVIEW-2026-07 #2）。掃除で容量を現実化してから

## Why この選択肢か

| 選択肢 | 採否 | 理由 |
|---|---|---|
| **A. 手動お掃除 + predecessor 無条件保護 + 期間基準（採用）** | ✅ | 容量問題の主犯（編集ありの古い workspace）に届く唯一の案。手動 + プレビュー + ごみ箱の三重の保険で法律実務データの誤削除リスクを最小化 |
| B. 「開いただけ」のみ削除 | ❌ | 62 個/数 MB しか回収できず 7.4 GB 問題を解決しない |
| C. 自動削除（起動時 N ヶ月超を掃除） | ❌ | 法律実務データにサイレント削除は不適。ユーザーが気づかないうちに「戻せる」状態が消えるのは ADR-0026 の趣旨に反する |
| D. 上限容量制（LRU で自動追い出し） | ❌ | C と同じサイレント性の問題 + 実装複雑 |

## Consequences

### 受け入れる trade-off

1. **削除した workspace の「編集可能な状態」は失われる**（ごみ箱から救出しない限り）。Dropbox 上のフラット PDF からの再取込は可能だが、overlay は編集不能な焼き込み済み。プレビューで編集有無を明示して判断材料を与える
2. **ごみ箱経由なのでディスクは即座には空かない**。結果表示に「ごみ箱を空にすると ○ MB 解放されます」の一文を添える
3. **predecessor 保護は「現存する workspace からの参照」のみ**が根拠。参照元 workspace 自体が削除候補（例: 確定版側も N ヶ月未アクセス）の場合、同一実行で両方消えるのは整合的（lineage ごと不要になった判断）。ただし**参照元が保持される限り predecessor も必ず保持**する
4. スキャンは全 `.kpdf3` の read-only open を伴うため 1,800 個規模で数十秒かかりうる → プログレス表示

### 影響範囲（想定）

- `src/main/workspace-cleanup.js`（新設）: 候補判定（純関数、`sidecar-sweep.js` と同じく依存なし・単体テスト可能な形）+ スキャン/削除実行
- `src/main/main.js` / `preload.cjs`: IPC（スキャン / 実行）、`shell.trashItem`
- `src/main/workspace-registry.js`: 削除行の DELETE ヘルパ追加
- `src/renderer/`: ツールメニュー項目 + 整理ダイアログ（98 風）
- `test/workspace-cleanup.test.mjs`（新設）: 保持判定のテーブル駆動テスト（predecessor 保護 / 開いただけ / N ヶ月境界 / 孤児 registry 行）

### 保存コア非干渉

読み取り + ファイル削除 + registry DELETE のみ。**保存形式・レイヤ構造・byte-copy ゲートには一切触れない**。

## 検証（実装時の完了条件 = REVIEW-2026-07 #1）

- 掃除実行後に (a) 既存案件が正常に開く (b) 確定版の「編集可能な状態に戻す」が生存 (c) `index.db` に削除済エントリが残らない
- predecessor 参照された workspace が N ヶ月超・未アクセスでも保持される
- 開いているタブの workspace が候補に出ない
- ごみ箱から `.kpdf3` を書き戻す → 再び開ける（registry は fingerprint 再登録で自己修復）
- `npm test` に保持判定テストが組み込まれ全 pass
